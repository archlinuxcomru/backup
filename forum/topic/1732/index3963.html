Вики я пользоваться умею, только вот русский вариант несколько отстаёт от оригинала. Короче, глаза боятся - руки делают.

1. /etc/default/grub - изменил строку до вида GRUB_CMDLINE_LINUX=“init=/bin/systemd”
2. grub-mkconfig -o /boot/grub/grub.cfg
3. reboot
Загрузилось. Чудеса.

4. pacman -S systemd-sysvcompat
на этом этапе облом. systemd-sysvcompat конфликтует с sysvinit, который не хочет удаляться, потому что от него зависит initscripts. Будем делать иначе.

4а. pacman -R initscripts && pacman -S systemd-sysvcompat
4б. Чёрт его знает, насколько это ещё нужно, но на всякий случай mv /etc/rc.conf.pacsave /etc/rc.conf && mv /etc/rc.local.pacsave /etc/rc.local
5. /etc/default/grub - изменил строку до вида GRUB_CMDLINE_LINUX=“”
6. grub-mkconfig -o /boot/grub/grub.cfg
7. reboot
загрузилось до экрана логина, gdm не грузится. Параллельно обнаружилось отсутствие русских шрифтов в консоли

8. systemctl enable gdm.service && systemctl start gdm.service
Вуаля, попали в иксы. Но. Сети нет, блутуса нет, прочего нет. Вручную активировал то, что ранее грузилось из секции DAEMONS /etc/rc.conf - syslog-ng, NetworkManager.service, bluetooth.service, sshd.service, smbd.service (с самбой по дефолту теперь не грузится nmbd, грузить руками или так сойдёт?).

9. reboot
Щастье, чистый systemd установлен.

10. внезапно выяснилось, что crond не запущен. Пытаюсь запустить, увлекательная переписка с systemctl ниже
[email protected]:~$ sudo systemctl is-active crond.service
Failed to issue method call^ Unit crond.service is not loaded
unknown
[email protected]:~$ sudo systemctl enable crond.service
Failed to issue method call: No such file or directory
[email protected]:~$ sudo systemctl start crond.service
[email protected]:~$ sudo systemctl is-active crond.service
active
[email protected]:~$ sudo systemctl is-enabled crond.service
disabled
[email protected]:~$ sudo systemctl enable crond.service
Failed to issue method call: No such file or directory
То есть загрузить руками можно, а заставить стартовать руками нельзя. И чо?

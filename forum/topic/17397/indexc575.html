И вопрос не по делу. Многие утилиты лезут в access("/etc/ld.so.preload", R_OK) = -1 ENOENT (No such file or directory). Которого конечно нету, как думаете стоит заводить preload?

Попробую ответить пошире ...
Для начала в общем о возвращаемом значении -1
Давал тебе выше ссылку на doc по системным вызовам — посмотри там возвращаемое значение и код ошибки.
Частенько в вызовах по открытию файлов возвращается значение -1, но это не значит, что это всегда является ошибкой — нужно смотреть конкретно — иногда программа ищет какой то файл по разным путям, не находит и ищет дальше, находит и успокаивается.
А вот если не находит, нужно смотреть, что за файл и нужен ли этот файл проге. И вот если бы этот файл был супер-пупер важным, то прога сразу бы заткнулась на конкретном системном вызове.
Насчет /etc/ld.so.preload — это файл, содержащий список разделённых пробелами динамических ELF библиотек, которые будут загружены перед программой. В нашей системе этот файл не нужен.
Практически он не используется, но иногда приходится. И частенько этот финт используют с помощью переменной окружения LD_PRELOAD, которая загружает определенную библиотеку до загрузки остальных. Мог заметить такой финт в топиках по играм, типа такого
LD_PRELOAD='/usr/$LIB/lib1.so /usr/$LIB/lib2.so' /usr/bin/game
UPD …. самое лучшее для понятия эксперимент — если у тебя этого файла нет, создай его, но пустой и проверь — это сообщение исчезнет, но в целом ни на что не повлияет.
У меня этот файл имеется и пустой (как появился, не знаю, то ли сразу был, то ли когда эспериментировал с динамическими библиотеками)
$ ls -la /etc/ld.so.preload
-rw-r--r-- 1 root root 0 ноя 7 17:53 /etc/ld.so.preload
$ file /etc/ld.so.preload
/etc/ld.so.preload: empty

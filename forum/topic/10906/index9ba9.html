Проблема, похоже, уже видна:
19:24:33.801879 IP 192.168.1.100.33757 > 178.45.249.163.80: Flags [S], seq 2284593979, win 5360, options [mss 536,sackOK,TS val 34542758 ecr 0,nop,wscale 7], length 0
19:24:33.857315 IP 178.45.249.163.80 > 192.168.1.100.33757: Flags [S.], seq 3001634960, ack 2284593980, win 5792, options [mss 1400,sackOK,TS val 3846079526 ecr 34538912,nop,wscale 6], length 0
19:24:33.857386 IP 192.168.1.100.33757 > 178.45.249.163.80: Flags [R], seq 2284593980, win 0, length 0
Видим, что идет запрос SYN, обратно получаем SYN/ASK и сразу посылаем RST. И так по кругу...
Если приглядимся, то видим, что метки timestamp (TS val 34542758 (туда) и ecr 34538912 (обратно)) не совпадают и линукс (по умолчанию в линуксе проверка timestapm включена) сбрасывает соединение.
По всей видимости, Волгателеком использует какой-то кривой файрвол, который создает подобные проблемы. В подобных случаях можно попробовать отключить эту фичу в ядре
динамически
echo 0 > /proc/sys/net/ipv4/tcp_timestamps
или создать файл /etc/sysctl.d/hell-volgatelecom.conf для настройки параметра при загрузке с такой строчкой
net.ipv4.tcp_timestamps = 0

PS: вот как раз попалась статья с вменяемым объяснением тех.деталей. Судя по ней, еще могут возникнуть проблемы и с TCP window scaling. Так что в Вашем случае можно сразу и
echo 0 > /proc/sys/net/ipv4/tcp_window_scaling
смело попробовать.

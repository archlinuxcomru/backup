Судя по логам причина в рендеринге, которым занимается программный рендер swrast, использующий для этого библиотеки libGL, посмотреть которые, включая и симлинки на эти библиотеки, можно командой
find /usr -iname "*libGL.so*" -exec ls -l -- {} +
привожу свой вывод
lrwxrwxrwx 1 root root     10 июн 26 05:13 /usr/lib/libGL.so -> libGL.so.1
lrwxrwxrwx 1 root root     14 июн 26 05:13 /usr/lib/libGL.so.1 -> libGL.so.1.7.0
-rwxr-xr-x 1 root root 538368 июн 26 05:13 /usr/lib/libGL.so.1.7.0
lrwxrwxrwx 1 root root     10 июн 26 05:14 /usr/lib32/libGL.so -> libGL.so.1
lrwxrwxrwx 1 root root     14 июн 26 05:14 /usr/lib32/libGL.so.1 -> libGL.so.1.7.0
-rwxr-xr-x 1 root root 435384 июн 26 05:14 /usr/lib32/libGL.so.1.7.0
При обновлении эти симлинки могут нарушиться, так что сравни свои выводы с этим. Также рекомендуют проверить это, запустив команду
export LIBGL_DEBUG=verbose; glxgears
если будут ошибки, то эти проблемные библиотеки должны быть указаны в выводе.

Дополнительно можно сделать и такие проверки
- все ли нормально с динамическими библиотеками
ldd /usr/bin/glxinfo
- какие билиотеки, связанные с OpenGL, libGL, libEGL, находятся в кэше (возможно это и лишнее, но проверить можно)
ldconfig -p | grep -i gl.so
привожу для сравнения свой вывод, убрав лишнее
        libQt5OpenGL.so.5 (libc6,x86-64) => /usr/lib/libQt5OpenGL.so.5
	libQt5OpenGL.so (libc6,x86-64) => /usr/lib/libQt5OpenGL.so
	libOpenGL.so.0 (libc6,x86-64) => /usr/lib/libOpenGL.so.0
	libOpenGL.so.0 (libc6) => /usr/lib32/libOpenGL.so.0
	libOpenGL.so (libc6,x86-64) => /usr/lib/libOpenGL.so
	libOpenGL.so (libc6) => /usr/lib32/libOpenGL.so
	libGL.so.1 (libc6,x86-64) => /usr/lib/libGL.so.1
	libGL.so.1 (libc6) => /usr/lib32/libGL.so.1
	libGL.so (libc6,x86-64) => /usr/lib/libGL.so
	libGL.so (libc6) => /usr/lib32/libGL.so
	libEGL.so.1 (libc6,x86-64) => /usr/lib/libEGL.so.1
	libEGL.so.1 (libc6) => /usr/lib32/libEGL.so.1
	libEGL.so (libc6,x86-64) => /usr/lib/libEGL.so
	libEGL.so (libc6) => /usr/lib32/libEGL.so

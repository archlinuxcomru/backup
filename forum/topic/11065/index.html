Есть.
menuentry 'Arch GNU/Linux, with Linux core repo kernel' --class arch --class gnu-linux --class gnu --class os $menuentry_id_o$
        load_video
        set gfxpayload=keep
        insmod gzio
        insmod part_msdos
        insmod ext2
        set root='hd1,msdos3'
        if [ x$feature_platform_search_hint = xy ]; then
          search --no-floppy --fs-uuid --set=root --hint-bios=hd1,msdos3 --hint-efi=hd1,msdos3 --hint-baremetal=ahci1,msdos3 $
        else
          search --no-floppy --fs-uuid --set=root b07b6171-80b2-4b80-acda-965998af3524
        fi
        echo    'Loading Linux core repo kernel ...'
        linux   /boot/vmlinuz-linux root=UUID=b07b6171-80b2-4b80-acda-965998af3524 ro
        echo    'Loading initial ramdisk ...'
        initrd  /boot/initramfs-linux.img

На некоторых (старых) материнках действительно есть проблема опознания БОЛЬШИХ ФЛЕШЕК в BIOS, функциями которого пользуется любой загрузчик для доступа к ДИСКАМ.
Разделы загрузчик распознает на диске сам, поскольку используемые им функции BIOS INT 13h предполагают работу на уровне секторов диска.
Проблема либо с доступом к слишком большой флешке вообще, либо к её секторам с большими номерами. Первое решается только флешкой меньшего размера, второе решается созданием в начале флешки достаточно малого раздела.

Что касается USB-винчестеров – не пробовал, но догадываюсь, что тут всё зависит от реализации в конкретном BIOS работы с ними по образцу ATA-диска или по образцу флешки.
ATA-диски с большими разделами на тех же материнках читаются совершенно нормально.

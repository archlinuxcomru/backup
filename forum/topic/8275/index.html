не пингуется машина с Arch Linux

heaven
# 13 лет, 2 месяца назад
Темы: 1
Сообщения: 7

Здравствуйте, уважаемые!

Перешерстил форумы “Сети, Серверы, Защита” и “Новичкам в Arch Linux”, но ничего по своей проблеме не нашел. Если пропустил - ткните, плиз, носом, буду благодарен.
Изучил HowTo http://wiki.archlinux.org/index.php/Configuring_network и http://wiki.archlinux.org/index.php/Настройка_сети_(Русский), все сделал по ним, ан нет, не выходит каменный цветок…

Стоит следующая задача: есть ноутбук с Windows Vista, он подключен к интернету через wimax (yota). К нему по кросс-кабелю ethernet подключен Intel Atom 330 на матери Intel D945GCLF2. На ноутбуке расшарен доступ к интернету для Atom (для локальной сети).
Для Atom назначается ip 192.168.0.9, у ноутбука на этот интерфейс ip 192.168.0.1
Если на Atom установить OpenSUSE 11.1 или xubuntu 9.04, то он ходит через ноутбук в интернет сразу после установки, качает обновления и т.д. Ноутбук и Atom друг друга пингуют. С ноутбука без проблем захожу на Atom по ssh и vnc.

После установки Arch Linux на Atom столкнулся с проблемой, что даже компы между собой не пингуются.

В Arch Linux на Atom в /etc/rc.conf прописано:
lo="lo 127.0.0.1"
eth0="eth0 192.168.0.9 netmask 255.255.255.0 broadcast 192.168.0.255"
INTERFACES=(lo eth0)
gateways="default gw 192.168.0.1"

После перезагрузки вижу следующее по route -n:
192.168.0.0 0.0.0.0 255.255.255.0 U 0 0 eth0
0.0.0.0 192.168.0.1 0.0.0.0 UG 0 0 eth0

При этом нет loopback-интерфейса. И не пингуются машины.
Atom под Arch Linux выдает на ping 192.168.0.1 следующее:
From 192.168.0.9 icmp_seq=N Destination Host Unreachable

А ноутбук под вистой выдает в свою очередь:
C:\>ping 192.168.0.9
Обмен пакетами с 192.168.0.9 по с 32 байтами данных:
Ответ от 192.168.0.1: Заданный узел недоступен.
Превышен интервал ожидания для запроса.
Превышен интервал ожидания для запроса.
Превышен интервал ожидания для запроса.
Статистика Ping для 192.168.0.9:
    Пакетов: отправлено = 4, получено = 1, потеряно = 3
    (75% потерь)

При этом ifconfig на Atom показывает, что оба интерфейса нормально сконфигурены:
eth0 inet addr:192.168.0.9 Bcast:192.168.0.255 Mask:255.255.255.0
lo inet addr:127.0.0.1 Mask:255.0.0.0
Оба UP RUNNING

Для проверки переустановил xubuntu, она “иcкаропки” качает пакеты обновлений, firefox без проблем работает, все пингуется.
На всякий пожарный смотрю iptables, они пустые, как и на Arch Linux.
Запоминаю route -n на xubuntu:
192.168.0.0 0.0.0.0 255.255.255.0 U 0
169.254.0.0 0.0.0.0 255.255.0.0 U 1000
0.0.0.0 192.168.0.1 0.0.0.0 UG 100

Снова точно так же, как и в предыдущий раз, ставлю Arch Linux, настраиваю /etc/rc.conf, после загрузки удаляю всё:
route del -net 192.168.0.0/24
route del default

Проверяю, route -n показывает пустую таблицу. Ввожу, то, что было на xubuntu и работало, прямо с метриками:
route add -net 192.168.0.0/24 metric 0 eth0
route add -net 169.254.0.0/16 metric 1000 eth0
route add default gw 192.168.0.1 metric 100 eth0

Проверяю по route -n - идентично, как на xubuntu.

Ноут теперь показывает:
C:\Windows>ping 192.168.0.9
Обмен пакетами с 192.168.0.9 по с 32 байтами данных:
Ответ от 192.168.0.1: Заданный узел недоступен.
Ответ от 192.168.0.1: Заданный узел недоступен.
Ответ от 192.168.0.1: Заданный узел недоступен.
Ответ от 192.168.0.1: Заданный узел недоступен.
Статистика Ping для 192.168.0.9:
    Пакетов: отправлено = 4, получено = 4, потеряно = 0
    (0% потерь)

Atom под Arch Linux выдает то же самое:
From 192.168.0.9 icmp_seq=N Destination Host Unreachable

На ноутбуке честный print route:
      192.168.0.0    255.255.255.0         On-link       192.168.0.1    276
      192.168.0.1  255.255.255.255         On-link       192.168.0.1    276
    192.168.0.255  255.255.255.255         On-link       192.168.0.1    276
[skip]
  255.255.255.255  255.255.255.255         On-link       192.168.0.1    276
Опять же под xubuntu и OpenSUSE с такими же настройками на ноутбуке все работает по дефолту.

Буду очень признателен, если кто-нибудь поможет разобраться с данной проблемой.

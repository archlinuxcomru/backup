спасибо, помогло, разобратся )

.htaccess содержит реврайт на www.aaaa.ru и www.bbb.ru соответственно.

сделал

ServerName www.aaaa.ru
ServerAlias aaaa.ru *.aaaa.ru

и

ServerName www.bbb.ru
ServerAlias bbb.ru *.bbb.ru

заработало, собсно до правки aaaa.conf работало как и написанно в документации

Now when a request arrives, the server will first check if it is using an IP address that matches the NameVirtualHost. If it is, then it will look at each <VirtualHost> section with a matching IP address and try to find one where the ServerName or ServerAlias matches the requested hostname. If it finds one, then it uses the configuration for that server. If no matching virtual host is found, then the first listed virtual host that matches the IP address will be used.

Думал думал про костыли .
Для работы с прокси сервером на интерфейсе с 192.168.0.0/24 не нужен df gw если прокси сервер находится в одном сегменте сети. Если в другом - нужны маршруты статические.
Имена при работе с прокси-клиентами разрешаются на прокси сервере (насколько я помню поведение по умолчанию броузеров)
Тобишь если ходить в инет через броузер - будут использоватся днс разрешение силами прокси сервера.

По идее можно сделать мегакостыль!

Обеспечить приоритет старта интерфейса с 192.168.12.0/24
После старта интерфейса в post.up скрипт добавить строку заменяющую dns сервер провайдера на dns локальный и правильный doman search.

На интерфейсе с 192.168.0.0/24, убрать df gw и обеспечить доступ к прокси серверу статическим маршрутом (если это нужно), броузеры настроить как клиенты прокси сервера.

Повесить скрипт проверяющий работу DNS сервер локального, в случае его затупа слазить в /etc/resolv.conf и поменять на dns провайдера.

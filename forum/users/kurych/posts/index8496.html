Навскидку, если меня внимательность не подвела, я бы Ваш скрипт в месте обработки цепочки FORWARD исправил как-то так:
======cut=======
### $IPT -N VERIFY_IP_MAC # verify (ip & MAC) address

INFORM_IN_USERS=/etc/iptables/users #“svyazka.ip-MAC.users.txt”

### if ; then
### $IPT_INPUT –in-interface $INT_DEV –source $INT -j VERIFY_IP_MAC
### $IPT_FORWARD –in-interface $INT_DEV –source $INT -j VERIFY_IP_MAC

(cat $INFORM_IN_USERS; echo) | # make sure there is a LF at the end
while read IP MAC GROUP LOCALISE; do
#
#IP MAC GROUP LOCALISE
#
#192.168.0.14 00:50:8D:51:67:F9 ..
#192.168.0.103 00:13:D4:65:7F:B0 ..
#
case “$IP” in
\#*|“”) continue ;;
esac

if ; then
### $IPT -A VERIFY_IP_MAC –source $IP –match mac ! –mac-source $MAC -j DROP
$IPT -A FORWARD –source $IP –match mac –mac-source $MAC -j ACCEPT
fi

if ; then
cmd_string=“–in-interface lo –source $IP”

else
cmd_string=“–in-interface $INT_DEV –source $IP”
fi

$IPT_INPUT $cmd_string -j $GROUP
$IPT_FORWARD $cmd_string -j $LOCALISE
done

======cut end=======

Давайте еще вот так посмотрим, попадает ли в эти правила что-нибудь:
sudo iptables -L –line-numbers -v
Но, в принципе, можно предположить, что в правиле
-A VERIFY_IP_MAC -s 192.168.0.102/32 -m mac ! –mac-source 00:50:FC:97:DD:4F -j DROP при несовпадении ip адреса второе условие по mac проверяться уже не будет, продолжит проверяться основная ветка правил и сработает одно из следующих, разрешающих либо icmp, либо на определенные порты.

То есть надо проверять в одном правиле оба параметра на соответствие для разрешения форвардинга, а не так, как Вы делаете.
